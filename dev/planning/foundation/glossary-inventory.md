## Methodology

This inventory was produced by reading every `.md` file under `/Users/ilya/ai-assistant/projects/front-knowledge-base/` outside `.obsidian/`, `Assets/`, and `.planning/codebase/` (16 files total: 4 root foundations, 2 decision-loop docs, 5 JTBDs, 3 user flows, 1 README, 1 CLAUDE.md), the existing definitions in `Basic info and definitions.md` and `Personas and audience.md`, and the curator-side parameter list in `/Users/ilya/ai-assistant/projects/docs-knowledge/new-docs-about/reference/risk-configuration-dictionary.md`. Candidate concepts were filtered to terms used 3+ times, capitalized as named concepts, italicized/bolded as terminology, or used as prefixes to documented field references; arbitrary descriptive prose was excluded. Each candidate was tagged with where it is currently defined (Personas / Basic info / risk-config-dict / nowhere) and whether it is foundation-relevant — i.e. a Pool LP or CA operator's decision changes if the concept changes, or the user must understand the actor to evaluate the position. Field references (`Type.field` form) are tracked separately in section (c) since they belong in a data schema, not the glossary. Section (d) lists terms whose canonical form, scope, or relationship to the risk-configuration-dictionary needs explicit reconciliation before glossary stubs are written.

## (d) Inconsistencies / canonicalisation candidates

The following terms have collisions, name drift, or scope ambiguity that need to be reconciled before glossary stubs are written. They are ordered by impact.

1. **"Quota" (vault) vs "Collateral-specific rate" + "Collateral limit" (risk-config-dict).** The vault speaks of a "quota system" with a "quota rate" and "quota composition"; the risk-config-dict splits the same machinery into two separate parameters: `Collateral-specific rate` (the APR component) and `Collateral limit` / `Quota Limit` (the per-collateral cap, where the dict itself notes "Quota Limit" as the alternate name). Decision needed: is "quota rate" = `Collateral-specific rate`? Is "per-token quota" the limit or the rate? The Basic-info entry should explicitly map `quota` → `Collateral-specific rate` + `Collateral limit` and pick a canonical user-facing label (likely keep "quota" as the user-facing umbrella term and reference both dict parameters underneath).

2. **HF vs Health Factor vs health factor.** Used in all three forms across the vault. CLAUDE.md uses "HF". JTBD docs mix "HF" and "health factor"; the Tim addendum to CA management JTBD always uses "health factor" lowercase. Pick one canonical form (recommendation: `Health Factor` with `HF` as documented alias) and standardise inside the glossary entry; do not retroactively rewrite the docs unless asked.

3. **"Credit Manager" vs "CM" — counterparty vs pool-side concept?** A CM is currently described two ways in the vault: (a) a "separate risk envelope" inside a pool (`User flows/Pool deposit - User flow (LP).md:46`) — pool-side concept; (b) a strategy template the CA operator picks (`User flows/Credit Account - User flow (CA operator).md:34`, "minDebt/maxDebt", "borrowable liquidity per CM") — counterparty/lifecycle concept. Glossary entry should hold both views in one definition; otherwise the LP and CA flows will pick up subtly different mental models.

4. **Session-mode taxonomy: 3-row vs 6-row.** `Decision-making loop/Entry points.md` defines three session modes (Decision / Monitoring / Emergency). `JTBDs/Multi-position & portfolio-level - JTBD.md` defines six (Confirmation / Analysis / Action / Emergency / Exit / Reallocation). CLAUDE.md acknowledges the latter is "the most refined version". Decide whether the glossary entry pins the 3-row taxonomy (and treats the 6-row matrix as a sub-classification of Monitoring entries) or the 6-row.

5. **"Composite APY" vs "Total APY" vs "Net APY" — overlap.** `composite total APY` appears in the LP user flow; `composite APY` in benchmarks; `net APY` in CA management; `total APY` in `Pool deposit User flow`. They are not the same thing: composite/total = organic + incentive (gross of LP fees); net = composite − fees − borrow − quota. Glossary should list the three explicitly and pin which one is the "headline" for each persona (LP headline = composite, CA headline = net).

6. **"Forbidden token" — single concept or two?** `forbidden-tokens mask` (CA-side: a held token becomes forbidden, triggers safe pricing) and `Collateral's forbidden status` (risk-config-dict parameter, curator switch) are the same protocol primitive but the vault and dict use different names. Standardise: `Forbidden token` with both the dict parameter and the user-side mask consequence in one entry.

7. **"Concentration cap" — user-side internal vs pool-side organic.** The vault uses "concentration cap" two ways: the user's internal cap (their own diversification rule, see `JTBDs/Pool deposit — JTBD.md:44`) and the pool-side concentration question (no single CM dominates total debt, see `:39`). Same word, two distinct concepts — glossary should label them separately ("user concentration cap" vs "pool concentration measure").

8. **"Quota composition" (LP-side) vs "per-token quota" (CA-side).** The LP-facing "quota composition" measures how borrower exposure mixes per token across the pool; the CA-facing "per-token quota" is the operator's own exposure to a quoted token via their CA. Same word, different scopes — flag in the glossary entry.

9. **"Borrow rate" vs "borrow APY" vs "borrow cost".** Used interchangeably (`borrowApy` is also a field name). In CA management, "borrow cost" is borrow rate × principal in absolute terms. Glossary should pick one canonical form.

10. **"Adapter" — counterparty or just a parameter?** Listed as a counterparty/actor in this inventory because adapter choice constrains the universe of routes the CA operator can use. Risk-config-dict treats `List of allowed adapters` as a curator-side switch only. Glossary should bridge the two: parameter from the dict + user-side consequence (route choice).
